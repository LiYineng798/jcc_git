# 首页“实时阵容排行”接入设计

## 1. 目标

在当前主站首页新增一个 Tab：`实时阵容排行`。

这个专区不接入现有用户阵容库，不走当前 `lineups` 表，也不影响普通用户上传的阵容列表。  
它只展示一套由外部上传的、已经整理好的 `team_codes_by_tier.verify.json` 数据。

用户在首页切换到 `实时阵容排行` 后，可以：

- 按 `S / A / B / C / D` 分组查看实时阵容
- 每组按分页浏览，每页展示 5 条
- 直接复制阵容码

当前阶段不支持：

- 实时阵容详情页
- 实时阵容搜索
- 实时阵容收藏、点赞、举报
- 后台管理页面直接维护实时阵容
- 服务端自动抓取 DataTFT

## 2. 范围与约束

### 2.1 范围内

- 首页新增 `实时阵容排行` Tab，位置在 `最新` 左边
- 新增独立实时阵容 API
- 新增独立上传接口
- 新增本地 Python 上传脚本
- 前端按 DataTFT 卡片风格复刻实时阵容卡片，但页面外壳继续沿用当前站点主题

### 2.2 范围外

- 不修改现有普通阵容的数据库结构
- 不把实时阵容写入现有 `lineups` 表
- 不新增实时阵容详情页
- 不接 DataTFT 直连抓取到主站线上服务
- 不在管理员后台中暴露实时阵容维护页面

### 2.3 用户已确认的约束

- 首页新增专区，不替换原阵容列表
- 上传格式固定为 `team_codes_by_tier.verify.json`
- 上传鉴权采用独立 `upload_token`
- 上传后按整包覆盖处理
- 前端按 `S / A / B / C / D` 分组
- 每组每页展示 5 个，其他通过翻页查看
- 卡片只提供复制按钮，不提供详情页

## 3. 现有系统接入点

当前首页由以下文件驱动：

- `templates/index.html`
- `static/app.js`
- `static/styles.css`

当前普通阵容列表通过：

- `GET /api/lineups`

返回普通用户阵容数据，并支持：

- 最新 / 最热 / 上升 / 推荐 / SS
- 我的收藏
- 我的阵容

本次新增的 `实时阵容排行` 不能混入这条链路，否则会污染：

- 普通阵容权限模型
- 点赞/收藏/举报逻辑
- 用户数据统计
- 管理员后台阵容管理语义

因此本次采用独立内容源 + 独立接口。

## 4. 设计方案

### 4.1 推荐方案

采用“独立 JSON 仓储 + 独立 API + 首页新专区”的方案。

核心思路：

1. 服务端单独保存一份实时阵容 JSON 文件
2. 首页新增 `实时阵容排行` Tab
3. 点击这个 Tab 后，前端改为请求实时阵容接口
4. 实时阵容卡片单独渲染，不复用普通用户阵容卡片动作区
5. 由你本地运行 Python 脚本，把整理好的 `team_codes_by_tier.verify.json` 上传到服务端

### 4.2 不选其它方案的原因

#### 不选“写入独立 SQLite 表”

虽然以后会更便于搜索、审计和后台维护，但这次需求明确是：

- 本地上传
- 整包覆盖
- 首页专区展示

因此文件型仓储更简单、更稳、改动更小。

#### 不选“混入现有 `lineups` 表”

因为会造成语义混乱：

- 实时阵容不是用户上传
- 不需要用户权限、编辑、举报、隐藏
- 不应出现在“我的阵容”“收藏”“后台阵容管理”里

## 5. 数据模型

### 5.1 输入格式

服务端接收你上传的完整 `team_codes_by_tier.verify.json`。

期望保留的核心结构：

```json
{
  "meta": { ... },
  "tiers": {
    "S": [ ... ],
    "A": [ ... ],
    "B": [ ... ],
    "C": [ ... ],
    "D": [ ... ]
  }
}
```

每条卡片至少应包含：

- `id`
- `title`
- `tier`
- `jccCode`
- `mainAvatar`
- `heroImages`

可保留但当前首页阶段不必全部使用：

- `bgImage`
- `heroId`
- `meta.selectedProxy`

### 5.2 服务端落盘格式

服务端保存上传后的完整 JSON，不做二次拆表。

建议在 `instance/` 或专用数据目录下保存，例如：

- `instance/live-comps.json`

同时保留一份上一次备份，例如：

- `instance/live-comps.previous.json`

### 5.3 服务端读取后的分页模型

接口返回时，按单个段位分页。

响应模型：

```json
{
  "tier": "S",
  "items": [ ... ],
  "total": 18,
  "page": 1,
  "page_size": 5,
  "total_pages": 4
}
```

另外提供一份分组摘要：

```json
{
  "tiers": [
    { "tier": "S", "total": 18 },
    { "tier": "A", "total": 26 },
    { "tier": "B", "total": 21 },
    { "tier": "C", "total": 9 },
    { "tier": "D", "total": 4 }
  ],
  "updated_at": "...",
  "source_meta": { ... }
}
```

## 6. API 设计

### 6.1 `GET /api/live-comps/summary`

用途：

- 首页切换到 `实时阵容排行` 后先加载总览
- 渲染分组标题
- 渲染每组总条数
- 展示最后更新时间

返回：

- 每个段位的总数
- 当前数据更新时间
- 可选的数据来源说明

### 6.2 `GET /api/live-comps`

请求参数：

- `tier`：`S / A / B / C / D`
- `page`：正整数，默认 `1`
- `page_size`：默认固定 `5`，服务端也可强制只接受 `5`

返回：

- 该段位当前页数据
- 总数
- 页码
- 总页数

### 6.3 `POST /api/live-comps/upload`

用途：

- 接收你本地上传的完整 JSON

鉴权：

- 通过独立上传密钥 `upload_token`

建议方式：

- Header：`X-Upload-Token: <token>`

请求体：

- 完整 `team_codes_by_tier.verify.json`

处理逻辑：

1. 校验 Token
2. 校验 JSON 结构
3. 先写临时文件
4. 校验通过后原子替换正式文件
5. 备份旧文件
6. 返回导入结果摘要

返回结果示例：

```json
{
  "ok": true,
  "tiers": {
    "S": 18,
    "A": 26,
    "B": 21,
    "C": 9,
    "D": 4
  },
  "total": 78
}
```

## 7. 首页前端设计

### 7.1 Tab 行为

首页 Tab 顺序调整为：

- `实时阵容排行`
- `最新`
- `最热`
- `上升`
- `推荐`
- `SS`
- `我的收藏`
- `我的`

其中：

- 点击普通 Tab：继续走现有 `/api/lineups`
- 点击 `实时阵容排行`：切换到实时专区模式

### 7.2 布局结构

实时专区仍在现有主列表区域中显示，不额外新开页面。

切换到该专区后，主内容区结构变为：

1. 实时专区顶部说明栏
   - 标题
   - 数据更新时间
2. 段位分组区
   - `S`
   - `A`
   - `B`
   - `C`
   - `D`
3. 每组下方显示当前页 5 条卡片和分页器

### 7.3 卡片内容

每张卡片展示：

- 主头像 `mainAvatar`
- 阵容名 `title`
- 段位 `tier`
- 英雄头像条 `heroImages`
- 阵容码 `jccCode`
- `复制阵容码` 按钮

不展示：

- 收藏
- 点赞
- 举报
- 编辑
- 删除
- 查看详情

### 7.4 视觉风格

整体页面继续使用当前主站主题：

- 简洁
- 白色主背景
- 现有的夜间 / 白天模式
- 现有字体体系

但实时阵容卡片本身参考你提供的 DataTFT 复刻样式：

- 左侧头像模块
- 右侧卡片主体
- 段位色条
- 英雄头像带

因此最终效果是：

- 页面壳子保持你当前站点风格
- 卡片视觉接近 DataTFT 热门阵容卡

### 7.5 复制行为

点击 `复制阵容码` 后：

1. 复制 `jccCode`
2. 给出当前站点统一 toast 提示

当前阶段不记录：

- 收藏
- 点赞
- 举报

可选项：

- 是否记录单独的实时阵容复制统计

当前默认不做，避免扩大范围。

## 8. 上传脚本设计

新增一个独立 Python 脚本，例如：

- `upload_live_comps.py`

用途：

- 由你本地手工运行
- 把完整 JSON 发到服务端

命令形式建议：

```bash
python upload_live_comps.py ^
  --file team_codes_by_tier.verify.json ^
  --url https://jcc.np5.top/api/live-comps/upload ^
  --token YOUR_UPLOAD_TOKEN
```

脚本职责：

1. 读取本地 JSON 文件
2. 本地做基础 JSON 解析检查
3. 发起 HTTP POST 上传
4. 输出成功/失败结果

上传脚本不负责：

- 自动抓取 DataTFT
- 自动生成 `team_codes_by_tier.verify.json`

这些继续由你现有抓取工具链负责。

## 9. 安全策略

### 9.1 上传接口鉴权

- 不复用前台登录态
- 不复用管理员页面会话
- 单独使用 `upload_token`

原因：

- 这是机器/脚本调用
- 使用成本低
- 与现有用户体系完全解耦

### 9.2 Token 存储

- 只存在服务端配置中
- 不出现在前端页面
- 不写入上传返回内容

### 9.3 JSON 校验

上传时至少校验：

- 顶层是对象
- 必须包含 `tiers`
- `tiers` 必须是对象
- `S/A/B/C/D` 对应值必须是数组（可为空）
- 每条卡片必须包含必要字段
- `heroImages` 必须是数组
- `jccCode` 必须是字符串

### 9.4 文件写入安全

采用以下顺序：

1. 写入临时文件
2. 验证可读
3. 备份旧文件
4. 原子替换正式文件

这样可避免：

- 上传中断导致正式文件损坏
- 部分写入造成首页读取失败

### 9.5 大小限制

应限制上传体积，避免误传异常大文件。  
当前阶段建议上限：

- `5 MB`

### 9.6 错误处理

上传失败时：

- 不覆盖旧文件
- 返回明确错误信息

前台读取失败时：

- 返回空列表和错误说明
- 首页展示“当前暂无可展示的实时阵容数据”

## 13. 最终结论

本次实现采用：

- 首页新增 `实时阵容排行` Tab
- 独立实时阵容 JSON 内容源
- 独立 `upload_token` 上传接口
- 你本地通过 Python 脚本手工上传
- 服务端整包覆盖保存
- 前端按 `S/A/B/C/D` 分组展示，每组每页 5 条
- 卡片仅支持复制阵容码

这是当前需求下改动最小、语义最干净、风险最低的实现方式。
