# 金铲铲阵容库开发协作部署流程

## 1. 目的

本文档用于固化当前项目的标准开发流程，避免以后出现以下问题：

- 在错误的目录里直接改代码
- 本地开发版本和 GitHub 版本不一致
- 服务器上线内容和本地确认版本不一致
- 上线时误操作数据库或跳过必要验证

后续默认按本文档执行，除非明确说明要临时变更流程。

---

## 2. 当前环境角色划分

### 2.1 `claude_project`

定位：

- 本地开发工作目录
- 新需求、修复、样式调整、页面改造，先在这里完成
- 本地验证、调试、人工查看，也先基于这里进行

原则：

- 正常情况下，所有业务代码修改先落在 `claude_project`
- 不把 `jcc_git` 当作首改目录

### 2.2 `jcc_git`

定位：

- Git 仓库目录
- 用于整理最终可提交版本
- 用于上传 GitHub
- 用于服务器 `git pull` 同步

原则：

- `jcc_git` 不作为默认开发主战场
- `jcc_git` 只承接已经在 `claude_project` 验证过的改动
- 从 `claude_project` 同步到 `jcc_git` 后，再进行提交和推送

### 2.3 GitHub

定位：

- 远程代码源
- 服务器更新代码的唯一标准来源

原则：

- 服务器不直接接收本地散文件覆盖上线
- 服务器优先通过 `git pull` 获取更新

### 2.4 服务器

定位：

- 线上运行环境
- 只负责拉取 GitHub 最新代码并部署

原则：

- 正常情况下不直接在服务器里手改业务代码
- 服务器的业务代码更新统一通过 `jcc-update` 脚本执行

---

## 3. 固定标准流程

以后任何正常需求开发，按下面顺序执行。

### 第一步：在 `claude_project` 开发

- 先阅读需求
- 在 `claude_project` 中修改代码
- 如涉及前端、后端、样式、模板、接口、测试，均优先改这里

禁止事项：

- 不要一开始就直接改 `jcc_git`
- 不要一开始就直接改服务器上的业务代码

### 第二步：在 `claude_project` 本地验证

根据需求复杂度，执行一种或多种验证：

- 启动本地服务查看页面效果
- 检查接口是否可用
- 检查手机端和电脑端布局
- 运行已有测试
- 检查数据库迁移是否正常

目标：

- 先确认 `claude_project` 是可用版本
- 避免把未验证代码同步到 Git 仓库

### 第三步：把改动同步到 `jcc_git`

- 将 `claude_project` 中确认好的改动同步到 `jcc_git`
- 同步时只同步本次需要交付的内容
- 保持 `jcc_git` 与 `claude_project` 的有效代码一致

重点：

- 同步后，以 `jcc_git` 作为准备提交的最终版本

### 第四步：在 `jcc_git` 做提交前检查

至少检查以下内容：

- 是否缺文件
- 是否多传无关文件
- 是否误带本地日志
- 是否误带数据库文件
- 文档是否需要更新
- 关键页面和接口是否与 `claude_project` 一致

数据库相关注意事项：

- `instance/`
- `*.sqlite3`
- `*.sqlite3-wal`
- `*.sqlite3-shm`

这些内容默认不提交到 Git。

### 第五步：上传到 GitHub

- 在 `jcc_git` 中执行 Git 提交
- 推送到 GitHub 远程仓库

要求：

- 推送前确认当前分支和目标仓库正确
- 推送内容必须来自已验证过的 `claude_project` 同步结果

### 第六步：服务器拉取并部署

服务器统一使用以下脚本更新：

`/usr/local/bin/jcc-update`

当前脚本职责：

- 更新前自动备份线上数据库
- 从 GitHub 执行 `git pull`
- 安装依赖
- 执行数据库迁移
- 重启 `jcc` 服务
- 重载 `nginx`
- 执行健康检查

原则：

- 不手工散改线上代码
- 优先跑标准更新脚本

### 第七步：上线后验证

部署完成后至少确认：

- 首页可打开
- 登录页可打开
- 管理页可打开
- 核心接口健康检查通过
- 本次修改的功能可正常使用

线上健康检查接口：

`https://jcc.np5.top/api/health`

## 4. 特殊说明

### 4.1 什么情况下允许直接改 `jcc_git`

只有以下情况可以考虑直接改 `jcc_git`：

- 当前需求只涉及 Git 仓库整理，不涉及业务开发
- 只是同步文档、说明文件、部署脚本
- 用户明确要求跳过 `claude_project`，直接在 `jcc_git` 改

除了上述情况，默认还是先改 `claude_project`。

### 4.2 什么情况下允许直接改服务器

一般仅限以下内容：

- `nginx` 配置
- `systemd` 服务配置
- 部署脚本
- HTTPS 证书与续期
- 数据库备份、恢复、替换

说明：

- 即使允许直接改服务器，也应尽量保留操作记录
- 如果改的是可版本化内容，后续要回写到本地仓库

### 4.3 数据库处理原则

当前线上数据库不通过 Git 管理。

原因：

- 数据库属于运行时数据，不应进入代码仓库
- 服务器拉取 GitHub 更新时，不应覆盖线上数据库

当前已采取的措施：

- 仓库已忽略数据库文件
- 服务器更新脚本在拉代码前自动备份数据库

线上数据库路径：

`/srv/jcc_git/instance/lineups.sqlite3`

线上数据库备份目录：

`/srv/jcc_backups/jcc`

## 5. 标准执行清单

每次正常开发默认按下面清单执行。

### 开发清单

1. 阅读需求
2. 在 `claude_project` 修改代码
3. 在 `claude_project` 本地验证
4. 同步改动到 `jcc_git`
5. 检查 `jcc_git` 内容是否完整
6. 提交并推送到 GitHub
7. 在服务器执行 `/usr/local/bin/jcc-update`
8. 完成上线后验证

### 禁止清单

- 不跳过 `claude_project` 直接开发 `jcc_git`
- 不把本地数据库提交进 Git
- 不把日志文件提交进 Git
- 不直接手改服务器业务代码作为常规上线方式
- 不未验证就推 GitHub
- 不未备份数据库就手工替换线上库

---

## 6. 推荐命名和协作习惯

### 6.1 改动描述

提交前应清楚说明：

- 改了什么
- 为什么改
- 改动范围在哪些文件
- 是否涉及数据库结构
- 是否需要服务器重启

### 6.2 数据库相关改动

如果需求涉及数据库：

- 先判断是纯数据替换，还是结构迁移
- 若只是内容替换，明确备份旧库再替换
- 若涉及结构变更，优先做兼容迁移逻辑，避免直接破坏旧数据

### 6.3 文档同步

以下情况应顺手更新文档：

- 新增页面
- 新增接口
- 修改部署方式
- 修改管理后台能力
- 修改开发协作流程

---

## 7. 一句话版本

后续默认流程是：

**先在 `claude_project` 开发和验证，再同步到 `jcc_git`，再推送到 GitHub，最后让服务器执行 `/usr/local/bin/jcc-update` 上线。**
