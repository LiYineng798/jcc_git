# 实时阵容复制统计接入设计

## 1. 目标

为首页 `实时阵容排行` 增加复制次数统计能力，并让管理员在后台独立查看这些统计结果。

本次只解决两件事：

- 用户点击实时阵容复制按钮后，服务端累计复制次数
- 管理员后台新增独立 Tab 查看实时阵容复制次数

## 2. 范围

### 2.1 范围内

- 新增实时阵容复制统计表
- 新增实时阵容复制接口
- 新增管理员实时阵容列表接口
- 后台新增 `实时阵容` 独立 Tab
- 支持按阵容名称 / `id` 搜索

### 2.2 范围外

- 不新增实时阵容详情页
- 不新增实时阵容点赞 / 收藏 / 举报
- 不做按天趋势分析
- 不改普通阵容复制统计逻辑

## 3. 设计决策

### 3.1 统计模型

采用聚合表方案，按 `live_comp_id` 记录总复制次数：

- `live_comp_id`
- `copy_count`
- `created_at`
- `updated_at`

原因：

- 当前需求只要求总复制次数
- 实时阵容源数据本来就是 JSON 文件，不需要再拆成业务表
- 聚合表实现最小，部署改动最小

### 3.2 关联键

复制统计绑定实时阵容 JSON 中的 `id`。

这意味着：

- 重新整包上传实时阵容时，只要 `id` 不变，复制次数会延续
- 如果某个 `id` 从新 JSON 中消失，后台默认不再展示它

### 3.3 后台展示方式

管理员后台新增独立 Tab：`实时阵容`

展示字段：

- 阵容名称
- 阵容 `id`
- 段位
- 复制次数
- 统计更新时间

排序规则：

1. 复制次数倒序
2. 同分时按当前实时阵容文件内顺序展示

## 4. 接口设计

### 4.1 `POST /api/live-comps/<live_comp_id>/copy`

用途：

- 在实时阵容复制成功后记录一次复制

行为：

- 若 `live_comp_id` 不存在于当前实时阵容 JSON，返回 `404`
- 存在则累计 `copy_count + 1`

响应：

```json
{
  "ok": true,
  "live_comp_id": "s-01",
  "copy_count": 3
}
```

### 4.2 `GET /api/admin/live-comps`

用途：

- 后台实时阵容工作台列表接口

请求参数：

- `q`：可选，按阵容名或 `id` 过滤
- `page`
- `page_size`

响应：

```json
{
  "items": [
    {
      "id": "s-01",
      "title": "四费主C阵容",
      "tier": "S",
      "mainAvatar": "https://...",
      "copy_count": 12,
      "copy_updated_at": "2026-05-15 14:20:00"
    }
  ],
  "total": 1,
  "page": 1,
  "page_size": 20,
  "total_pages": 1,
  "updated_at": "2026-05-15T14:00:00",
  "source_meta": {
    "is_valid": true
  }
}
```

## 5. 前端改动

### 5.1 首页

- 复制实时阵容时，先完成剪贴板写入
- 剪贴板写入成功后再调用复制统计接口
- 若统计失败，不影响用户已经复制成功

### 5.2 后台

- 新增 `实时阵容` Tab
- 默认进入该 Tab 时加载第一页数据
- 提供搜索框
- 支持分页

## 6. 验证

需要覆盖：

- 复制实时阵容后计数累加
- 无效 `live_comp_id` 返回 404
- 管理员接口能看到复制次数
- 管理员接口支持搜索
- 后台页面包含独立 Tab
