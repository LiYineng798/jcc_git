# 每日 UV 统计设计说明

## 1. 背景

当前项目后台已有：
- 今日注册数
- 今日登录数
- 用户、阵容、举报、审计等管理能力

但还没有“每天访问网页的用户有多少”的真实 UV（Unique Visitor，独立访客）统计。

用户希望统计口径覆盖：
- 游客
- 已登录用户

并且希望统计结果可以进入现有后台，便于直接查看，不依赖第三方分析平台。

---

## 2. 目标

实现站点级“每日 UV”统计，满足以下要求：

1. 统计范围包含游客与登录用户。
2. 统计对象是“访问网页页面的人数”，而不是 API 请求次数。
3. 同一人在同一天内重复访问，只计一次。
4. 统计结果进入现有后台管理数据接口 `/api/admin/stats`。
5. 数据存储在当前 SQLite 数据库中，保持项目自托管。
6. 为后续扩展到“页面级 UV”“7 日趋势”等能力保留结构空间。

---

## 3. 口径定义

### 3.1 UV 定义

“每日 UV”指：某个自然日内访问过网站页面的独立访客数量。

### 3.2 去重规则

采用如下优先级进行去重：

1. **登录用户**：按 `user_id + 日期` 去重
2. **游客（有 cookie）**：按 `visitor_token cookie + 日期` 去重
3. **游客（无 cookie）**：按 `IP + 日期` 去重

### 3.3 为什么这样定义

- 单独按 IP 统计误差较大：同网关多人会被合并，同一人换网络会被拆分。
- 登录用户按 `user_id` 最稳定。
- 游客使用浏览器持久 cookie 能明显提升准确度。
- cookie 不存在时再退回 IP，保证统计不会失效。

---

## 4. 不统计的内容

本次不纳入统计：

- `/api/*` 接口调用次数
- 静态资源请求（CSS / JS / 图片 / favicon）
- 机器人访问识别
- 地区、设备、来源渠道分析
- 实时在线人数
- 历史补算

本次目标仅聚焦：**站点每日 UV**。

---

## 5. 推荐方案

推荐采用：**项目内建 UV 统计方案**。

### 5.1 备选方案对比

#### 方案 A：项目内建 UV 统计（推荐）

做法：
- 在现有 Flask 应用中记录页面访问
- 写入 SQLite
- 后台直接展示 UV 数据

优点：
- 数据完全自管
- 与现有后台无缝集成
- 适合当前项目体量
- 后续容易扩展到 7 日趋势、页面级统计

缺点：
- 需要一次数据库结构扩展
- 需要少量后端与后台改动

#### 方案 B：Nginx/CDN 日志离线统计

做法：
- 使用 Nginx 或 CDN 日志分析访问量

优点：
- 对应用代码侵入小

缺点：
- 不方便在现有后台页面直接展示
- 难以区分登录用户与游客
- 后续统计逻辑会分散到系统外部

#### 方案 C：第三方统计平台

做法：
- 接入 Umami / Plausible / Google Analytics 等

优点：
- 图表和报表成熟

缺点：
- 引入外部依赖
- 数据在外部平台
- 与当前轻量自托管项目不完全匹配

### 5.2 结论

本项目采用 **方案 A** 最合适。

---

## 6. 数据模型设计

新增一张访问统计表，建议命名为：`visit_events`

### 6.1 表结构建议

字段：

- `id`：主键
- `visit_date`：访问日期，格式 `YYYY-MM-DD`
- `visitor_key`：去重主键值，例如：
  - `user:12`
  - `guest:5f3c...`
  - `ip:1.2.3.4`
- `visitor_kind`：访客类型，取值建议：
  - `user`
  - `guest_token`
  - `ip_fallback`
- `user_id`：若为登录用户则记录，否则为空
- `visitor_token`：游客 cookie 值，若使用 token 去重则记录，否则为空
- `ip_address`：访问来源 IP
- `page_key`：访问页面类型，建议记录：
  - `home`
  - `auth`
  - `admin`
  - `lineup_editor`
- `created_at`：首次记录时间

### 6.2 唯一约束

建议唯一键：

- `(visit_date, page_key, visitor_key)`

说明：
- 如果只想做“站点总 UV”，理论上 `(visit_date, visitor_key)` 也可以。
- 但保留 `page_key` 的唯一键结构，可以让后续拓展“页面级 UV”更自然。
- 后台统计“全站 UV”时，按当天对 `visitor_key` 做去重即可。

### 6.3 为什么不用直接对 `user_id / visitor_token / ip` 建多个唯一键

SQLite 在 `NULL` 上的唯一约束行为不利于这种多口径场景。

统一使用 `visitor_key` 字段有几个优势：
- 去重逻辑简单
- 查询逻辑统一
- 后续统计更容易维护

---

## 7. 访问记录触发点

### 7.1 统计页面访问，不统计 API

本次只在 HTML 页面路由触发记录，不在 API 路由记录。

原因：
- 用户一个页面会触发很多 API 请求
- 如果统计 API，会把“一个人访问一次页面”误膨胀为多次访问
- UV 的本质是“页面访问者”，不是“请求数”

### 7.2 建议纳入统计的页面

记录以下页面访问：

- `/` -> `home`
- `/auth` -> `auth`
- `/admin` -> `admin`
- `/lineup/new` -> `lineup_editor`
- `/lineup/<id>/edit` -> `lineup_editor`

### 7.3 不纳入统计的路由

- `/api/*`
- `/favicon.ico`
- `/static/*`
- 健康检查 `/api/health`

---

## 8. visitor_token 设计

### 8.1 cookie 作用

用于提升游客 UV 去重准确度。

### 8.2 生成时机

当用户访问页面时：
- 如果是登录用户，可以不依赖该 cookie 做最终去重
- 如果是游客且没有 `visitor_token`，则生成一个随机 token 并下发 cookie

### 8.3 cookie 建议属性

- 名称：`visitor_token`
- 有效期：180 天或 365 天
- `HttpOnly = true`
- `SameSite = Lax`
- `Secure = true`（线上 HTTPS 环境）

### 8.4 说明

- 登录用户最终仍按 `user_id` 计数
- cookie 主要服务于游客统计
- 即便 cookie 丢失，仍可回退到 IP 统计，不影响功能可用性

---

## 9. 服务端记录逻辑

建议增加一个统一访问记录函数，例如：

- `record_page_visit(page_key)`

### 9.1 输入

- `page_key`
- 当前用户（若已登录）
- cookie 中的 `visitor_token`
- 客户端 IP

### 9.2 visitor_key 生成规则

按优先级生成：

1. 登录用户：`user:<user_id>`
2. 游客有 token：`guest:<visitor_token>`
3. 游客无 token：`ip:<ip_address>`

### 9.3 入库逻辑

- 计算 `visit_date = 今天`
- 尝试写入 `visit_events`
- 若唯一键冲突，则忽略

这样可以保证：
- 同一人同一天同一页面不会重复计数
- 重复刷新页面不会制造脏数据

---

## 10. 后台统计展示

扩展现有 `/api/admin/stats` 返回内容。

### 10.1 当前已有字段

- `total_users`
- `today_users`
- `today_logins`
- `hourly_registrations`

### 10.2 新增字段建议

- `today_uv`：今天全站 UV
- `yesterday_uv`：昨天全站 UV
- `last_7_days_uv`：最近 7 天 UV 趋势数组

建议 `last_7_days_uv` 结构如下：

```json
[
  {"date": "2026-05-04", "uv": 12},
  {"date": "2026-05-05", "uv": 18},
  {"date": "2026-05-06", "uv": 16}
]
```

### 10.3 后台 UI 展示建议

在后台统计卡片区域新增：
- 今日 UV
- 昨日 UV

可选：
- 最近 7 日 UV 列表或小型文本趋势块

本次如果想控制改动规模，第一阶段只做：
- 今日 UV
- 昨日 UV
- `/api/admin/stats` 中返回 7 日趋势数据

前端图表可以后续再做。

## 11. 隐私与误差说明

### 11.1 隐私边界

当前项目本来已经记录了部分 IP（如登录事件、复制事件、限流等），因此新增 UV 统计在隐私模型上没有本质变化。

### 11.2 误差来源

以下情况会造成统计偏差：
- 游客清除 cookie
- 游客更换浏览器 / 设备
- 多人共用同一网络且 cookie 不可用时退回 IP 去重
- 同一用户既以游客身份访问，又在当天登录访问

### 11.3 对当前项目是否可接受

可接受。

对当前项目规模（约百人级访问，非高并发），该口径已经足够实用，明显优于单纯按 IP 统计。

## 12. 数据库迁移策略

本次需要数据库结构扩展。

### 12.1 迁移内容

- 在 `db.py` 的 schema 中新增 `visit_events`
- 在迁移逻辑中补充“旧库没有该表时自动创建”能力

### 12.2 历史数据

不补历史 UV。

说明：
- 当前系统过去没有访客页面访问记录
- 无法准确回溯历史 UV
- 从功能上线当天开始累计即可

---

## 13. 测试策略

需要新增自动化测试，覆盖以下行为：

1. **登录用户 UV 去重**
   - 同一用户同一天访问多次，只记 1 个 UV

2. **游客 token 去重**
   - 同一 token 同一天访问多次，只记 1 个 UV

3. **游客 IP 回退去重**
   - 无 token 时按 IP 去重

4. **登录优先级高于 cookie**
   - 登录用户应使用 `user:<id>` 作为 visitor_key

5. **后台统计接口返回 UV 字段**
   - `/api/admin/stats` 返回 `today_uv`、`yesterday_uv`、`last_7_days_uv`

6. **页面访问触发记录，API 不触发**
   - 打开页面会写记录
   - 调 `/api/lineups` 不会写 UV

---

## 14. 实施范围控制

第一阶段只做：

1. 新增 `visit_events` 表
2. 页面访问记录逻辑
3. 游客 `visitor_token` cookie
4. 后台 `/api/admin/stats` 增加 UV 数据
5. 后台统计卡片显示今日 UV / 昨日 UV
6. 自动化测试

明确不做：
- 图表系统
- 实时在线统计
- 访问来源分析
- 地域分析
- 设备分析
- 机器人过滤

---

## 15. 最终结论

本项目最合适的方案是：

**在现有 Flask + SQLite 架构内新增页面访问 UV 统计能力。**

具体口径：
- 登录用户：按 `user_id + 日期` 去重
- 游客：按 `visitor_token + 日期` 去重
- 无 token 时：按 `IP + 日期` 去重

具体落点：
- 只统计 HTML 页面访问
- 不统计 API 请求
- 数据写入 SQLite
- 后台 `/api/admin/stats` 直接输出 UV 数据

这是当前项目阶段下，准确度、复杂度、可维护性最平衡的方案。
