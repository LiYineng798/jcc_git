# 后台“实时阵容手动补码”设计

## 1. 目标

在现有后台“实时阵容”模块中，增加一个管理员维护能力：

- 管理员可以按赛季查看实时阵容条目
- 对“已经抓到、但当前没有阵容码”的实时阵容，管理员可以手动补录阵容码
- 手动补录的阵容码在后续重新上传该赛季实时阵容数据时默认保留
- 只有当新的上传数据里该条实时阵容明确带有新的阵容码时，才覆盖管理员之前补录的阵容码

本次能力只解决“给现有实时阵容条目补码”，不扩展为“后台新建实时阵容条目”。

## 2. 范围与约束

### 2.1 范围内

- 后台实时阵容支持按赛季切换查看
- 后台实时阵容列表支持分页查看当前赛季所有条目
- 仅对“当前没有阵容码”的条目显示“补码”按钮
- 管理员可在弹窗中录入阵容码并保存
- 读取实时阵容时，后端自动合并“原始实时阵容数据”和“管理员手动补码覆盖层”
- 上传新的赛季实时阵容时，自动处理补码保留/失效逻辑
- 审计日志记录管理员补码操作

### 2.2 范围外

- 不支持后台新建实时阵容条目
- 不支持手工清空阵容码
- 不支持对“原始数据里已经有阵容码”的条目强制改码
- 不修改普通用户阵容上传、编辑、复制逻辑
- 不改现有用户数据库结构

### 2.3 已确认约束

- 管理员只处理“已经抓到的实时阵容条目”
- 手动补码后，后续上传默认保留管理员补码
- 若后续上传里该条实时阵容明确给出了新的阵容码，则以上传内容为准
- 后台只对“当前没有阵容码”的条目显示补码按钮
- 管理员只能补码，不提供“手工删除/清空补码”的入口

## 3. 现有系统接入点

当前实时阵容能力主要由以下文件组成：

- `live_comps.py`
- `admin.py`
- `templates/admin.html`
- `static/admin.js`

当前赛季信息已经独立存在，并可通过后台接口维护：

- `GET /api/admin/live-comps/seasons`
- `PUT /api/admin/live-comps/seasons/<season_id>`

当前实时阵容上传与前台读取逻辑已经按赛季拆分：

- `GET /api/live-comps/seasons`
- `GET /api/live-comps/summary?season=...`
- `GET /api/live-comps?season=...`
- `POST /api/live-comps/upload?season=...`

当前后台“实时阵容”Tab 仅显示：

- 实时阵容整体复制统计
- 赛季状态管理

尚不支持：

- 按赛季查看条目列表
- 查看单条实时阵容是否缺码
- 对缺码条目手工补码

## 4. 方案对比

### 4.1 方案 A：直接改赛季实时阵容 JSON

管理员补码后，直接把对应赛季的实时阵容 JSON 中的 `jccCode` 写回原文件。

优点：

- 改动最少
- 读取链路简单

缺点：

- 原始抓取数据与人工修补数据混在一起
- 重新上传时必须做复杂合并
- 后续排查“这条码来自抓取还是人工补录”不清晰

### 4.2 方案 B：独立“手动补码覆盖层”

原始实时阵容 JSON 保持不动；另存一份“赛季 + 实时阵容 ID -> 手动补码”的覆盖数据。读取时合并，上传时对账。

优点：

- 原始数据与人工修补职责分离
- 不污染上传原文件
- 回滚简单
- 风险低，不需要数据库迁移
- 符合当前按赛季 JSON 落盘的实现方式

缺点：

- 需要新增一层合并逻辑
- 后台接口需要扩展

### 4.3 方案 C：数据库表存补码

新增 SQLite 表，保存赛季、实时阵容 ID、补码、操作人、时间等结构化数据。

优点：

- 可扩展性最好
- 后续更容易做历史版本、筛选、统计

缺点：

- 本次需求偏重
- 需要数据库迁移
- 对线上结构影响更大

### 4.4 推荐方案

采用 **方案 B：独立“手动补码覆盖层”**。

原因：

- 与当前“按赛季存实时阵容 JSON”的架构最匹配
- 不改数据库结构，不影响现有用户数据
- 方便后续重新上传赛季实时阵容时做保留/覆盖判断
- 风险最低，回滚最直接

## 5. 数据模型设计

### 5.1 原始实时阵容文件

继续沿用现有赛季实时阵容 JSON，不修改其基本结构：

```json
{
  "meta": {},
  "tiers": {
    "S": [
      {
        "id": "comp-1",
        "title": "阵容名称",
        "tier": "S",
        "mainAvatar": "/api/live-comps/assets/...",
        "heroImages": ["/api/live-comps/assets/..."],
        "jccCode": ""
      }
    ]
  }
}
```

### 5.2 手动补码覆盖文件

每个赛季一份覆盖文件，建议放在实时阵容数据目录旁边，例如：

- `instance/live-comps/manual-codes/s17-cosmic.json`
- `instance/live-comps/manual-codes/s16-legends.json`

建议结构：

```json
{
  "season": "s16-legends",
  "updated_at": "2026-05-24T16:30:00",
  "items": {
    "comp-1": {
      "jccCode": "##xxxxxxxx",
      "updated_at": "2026-05-24T16:30:00",
      "updated_by": 1
    }
  }
}
```

说明：

- `items` 用实时阵容 `id` 做 key，便于快速定位
- 只保存“人工补录”的条目
- 原始数据已有阵容码的条目不写入这里

### 5.3 读模型

后端返回给后台和前台时，增加以下衍生字段：

- `resolvedJccCode`：合并后的最终阵容码
- `hasCode`：当前最终是否有可用阵容码
- `codeSource`：`original / manual / none`

返回规则：

- 原始 `jccCode` 有值：`resolvedJccCode = 原始值`，`codeSource = original`
- 原始 `jccCode` 为空且人工补码存在：`resolvedJccCode = 人工补码`，`codeSource = manual`
- 两者都没有：`resolvedJccCode = ''`，`codeSource = none`

## 6. 读取与上传合并规则

### 6.1 读取规则

读取任一赛季实时阵容时：

1. 先读该赛季原始实时阵容 JSON
2. 再读该赛季手动补码覆盖文件
3. 对每个实时阵容条目按 `id` 做合并
4. 将合并结果返回给前台/后台

### 6.2 上传规则

当上传新的某赛季实时阵容包时：

1. 先校验上传 JSON
2. 上传过程中继续做图片本地化处理
3. 写入新的赛季原始实时阵容文件
4. 读取该赛季手动补码覆盖文件并做清理：
   - 若某条新上传数据现在已有 `jccCode`，则删除该条对应的人工补码记录
   - 若某条新上传数据依旧没有 `jccCode`，则保留该条人工补码记录
5. 保存清理后的覆盖文件

这样可以保证：

- 管理员补码默认保留
- 上传方显式提供了新阵容码时，系统自动以上传数据为准

## 7. 后台交互设计

### 7.1 实时阵容工作台布局

继续保留现有：

- 今日复制
- 累计复制
- 最近统计更新时间
- 赛季状态管理

并新增：

- 赛季筛选器
- 当前赛季实时阵容列表
- 分页器

### 7.2 实时阵容列表字段

每条记录至少展示：

- 段位
- 阵容名称
- 实时阵容 ID
- 当前阵容码状态
- 阵容码来源

建议展示文案：

- `原始阵容码`
- `管理员补码`
- `暂无阵容码`

### 7.3 补码按钮显示规则

仅当条目满足以下条件时显示“补码”按钮：

- 原始 `jccCode` 为空
- 合并后的 `resolvedJccCode` 也为空

因此：

- 原始已有阵容码：不显示按钮
- 已经被管理员补过码：不显示按钮
- 仍缺码：显示按钮

### 7.4 补码弹窗

点击“补码”后打开轻量弹窗，展示：

- 赛季名
- 段位
- 阵容名
- 实时阵容 ID
- 阵容码输入框

保存时：

- 复用现有 `extract_lineup_code` 规范化逻辑
- 校验格式无误后提交
- 成功后刷新当前赛季列表
- 当前条目按钮消失，状态变为“管理员补码”

## 8. API 设计

### 8.1 扩展 `GET /api/admin/live-comps`

新增请求参数：

- `season`：赛季 ID
- `page`：页码
- `page_size`：分页大小

返回结果除了现有整体统计外，还要带当前赛季分页列表，例如：

```json
{
  "updated_at": "2026-05-24T16:30:00",
  "today_copy_count": 12,
  "total_copy_count": 345,
  "copy_updated_at": "2026-05-24 16:28:00",
  "season": {
    "id": "s16-legends",
    "name": "S16·英雄联盟传奇"
  },
  "items": [
    {
      "id": "comp-1",
      "title": "阵容名称",
      "tier": "S",
      "jccCode": "",
      "resolvedJccCode": "",
      "hasCode": false,
      "codeSource": "none"
    }
  ],
  "total": 18,
  "page": 1,
  "page_size": 20,
  "total_pages": 1
}
```

该接口服务于后台“按赛季看实时阵容条目”的需求。

### 8.2 新增 `POST /api/admin/live-comps/<season_id>/<live_comp_id>/manual-code`

用途：

- 给缺码实时阵容补录阵容码

请求体：

```json
{
  "code": "【阵容码】##xxxxxxxx"
}
```

处理逻辑：

1. 校验管理员身份
2. 读取该赛季原始实时阵容
3. 定位 `live_comp_id`
4. 若条目不存在，返回 `404`
5. 若原始条目已带 `jccCode`，返回 `400`
6. 用 `extract_lineup_code` 解析并规范化输入
7. 写入该赛季手动补码覆盖文件
8. 记录审计日志
9. 返回该条合并后的最新状态

### 8.3 审计日志

记录动作建议为：

- `admin_add_live_comp_manual_code`

记录内容至少包含：

- 管理员 ID
- 赛季 ID
- 实时阵容 ID
- 阵容名称
- 操作前值
- 操作后值

## 9. 错误处理

### 9.1 后端错误

- 赛季不存在：`404`
- 实时阵容条目不存在：`404`
- 条目原始数据已经有阵容码，不允许补码：`400`
- 输入阵容码无法解析：`400`
- 覆盖文件写入失败：`500`

### 9.2 前端提示

后台文案应明确区分：

- `当前条目已有原始阵容码，无需补码`
- `阵容码格式无法识别，请检查后重试`
- `补码已保存`

## 10. 测试与验收

至少覆盖以下场景：

1. 后台切换赛季，能正确查看该赛季实时阵容分页列表
2. 缺码条目显示“补码”按钮
3. 原始已有码条目不显示“补码”按钮
4. 管理员补码成功后，该条目状态变为“管理员补码”
5. 前台复制该条实时阵容时，使用的是合并后的 `resolvedJccCode`
6. 同赛季重新上传且仍无新码时，管理员补码保留
7. 同赛季重新上传且带新码时，管理员补码自动失效
8. 不同赛季的补码互不影响

## 11. 风险与回滚

### 11.1 风险

- 若实时阵容 `id` 不稳定，管理员补码会失去关联基础
- 若上传方更换了条目 ID 生成逻辑，旧补码无法自动迁移

### 11.2 当前可接受性

在你现有流程里，实时阵容补码的前提就是“按同一个实时阵容 ID 做覆盖”。因此本次默认要求上传脚本保持 `id` 稳定。

### 11.3 回滚方式

若本次能力需要回滚：

- 删除后台补码入口
- 停用覆盖层读取逻辑
- 保留原始赛季实时阵容 JSON 不动

由于本方案不改数据库结构，因此回滚成本较低。
